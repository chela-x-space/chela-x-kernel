## K4.2 Scope
K4.2 adds pure in-memory runtime lifecycle control on top of K4.1 registration. It covers heartbeat validation, heartbeat freshness, lease assessment and renewal, runtime health derivation, runtime state snapshots, recovery eligibility, supervisor recommendations, and validated registry updates.

## Runtime Lifecycle Boundary
- K4.2 does not execute agents.
- K4.2 does not schedule supervision loops.
- K4.2 does not persist runtime state.
- K4.2 does not publish events.
- K4.2 evaluates caller-supplied state and returns deterministic outcomes only.

## Heartbeat Model
- K4.2 preserves the K4.1 heartbeat record and adds an additive observed-heartbeat constructor with agent identity, reported presence, reported health, active lease reference, and evidence.
- Validated heartbeat ingestion rejects runtime mismatch, agent mismatch, lease mismatch, timestamp regression, stale heartbeats, and retired runtimes.
- Validation precedence is deterministic: runtime binding, agent binding, terminal or retired prohibition, duplicate detection, timestamp regression, lease consistency, freshness classification, then accepted update.
- Duplicate heartbeat submission is deterministic and leaves registry state unchanged.

## Freshness Policy
- Freshness uses caller-supplied cutoffs, not a hidden clock or hard-coded interval.
- Policy contains `late_cutoff`, `stale_cutoff`, and a rule version.
- Assessment returns `Fresh`, `Late`, `Stale`, or `Missing`.

## Lease Model
- K4.2 preserves the K4.1 lease record and adds an additive governed lease issuance constructor with agent binding, renewal lineage, and evidence.
- Lease assessment evaluates runtime match, agent match, expiry, and missing state using caller-supplied time.
- Renewal is explicit, deterministic, and fail-closed when lineage or evidence is incomplete.
- Renewal validation distinguishes exact duplicates from sequence regression. An already-applied renewal returns `LeaseDuplicate`; a different lower or invalid lineage request returns `LeaseSequenceRegression`.
- Terminal or retired runtimes reject renewal before duplicate, sequence, or renewal-window checks can create a non-terminal classification.

## Runtime Health
- Runtime health remains separate from presence.
- Health is derived deterministically from heartbeat freshness, lease validity, failure evidence, and recovery state.
- Idle is not treated as failure.

## Supervisor Observation
- Supervisor consumes immutable runtime snapshots and produces observations without mutating registry state.
- Observation includes heartbeat assessment, lease assessment, recovery eligibility, and current snapshot facts.

## Supervisor Recommendation
- Recommendations are pure and non-executing.
- Supported actions are `NoAction`, `MarkLate`, `MarkUnresponsive`, `TransitionOffline`, `SuspendRuntime`, `RequestRecovery`, `RejectRecovery`, and `RetireRuntime`.
- Critical failure, stale or missing heartbeat, or invalid lease moves the recommendation toward isolation before continued privileged work.

## Failure And Recovery Eligibility
- Failure observations are explicit and evidence-backed.
- Recovery eligibility requires failure evidence, a recovery reference, non-terminal runtime state, and a valid lease assessment.
- Recovery does not erase failure history.

## Registry Operations
- K4.2 adds validated operations for `record_heartbeat_validated`, `renew_lease_validated`, `expire_lease`, `runtime_snapshot`, `supervise_runtime`, `apply_supervisor_outcome`, and `apply_presence_transition`.
- Operations clone and validate before replacing registry state, so rejected operations do not leave partial mutation behind.

## Deterministic Trace
- Supervisor outcomes include ordered trace steps with decisive-step markers and stable evidence references.
- The trace is pure data and has no logging dependency.

## Time Injection
- All evaluation receives `TimeReference` from the caller.
- Public runtime evaluation logic does not call `SystemTime::now()` or `Instant::now()`.

## Out-Of-Scope Execution
- background workers
- process control
- restart loops
- network transport
- event bus
- workflow execution
- agent execution

## Relationship To K4.1
- K4.1 established immutable registration, capability indexing, and baseline presence transitions.
- K4.2 consumes that registry and extends it with validated supervision logic only.

## Relationship To K4.3
- K4.2 prepares runtime snapshots, supervisor traces, and deterministic outcomes that K4.3 can later project into event records.
- K4.2 does not introduce an event bus or event stream.

## Deferred Infrastructure
- persistent runtime state storage
- distributed supervision
- heartbeat transport
- automatic restart or recovery execution
- background expiration processing

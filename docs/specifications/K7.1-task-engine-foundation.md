## Purpose
Define the foundational K7 task vocabulary and identity model without introducing execution behavior or infrastructure.

## Scope
- Stable task identity and reference vocabulary
- Foundational task validation outcomes
- Additive reuse of frozen K1-K6 domain primitives
- Deterministic representation of task entities without persistence or runtime handles

## Non-Goals
- Task execution
- Scheduler behavior
- Worker or executor runtime
- Persistence, repositories, or databases
- Event publication or event bus integration
- Networking, REST, or RPC
- Async runtime or process management

## Terminology
- `TaskDefinitionId`: stable identity for one task-definition family.
- `TaskInstanceId`: stable identity for one concrete governed task occurrence.
- `TaskDependencyId`: stable identity for one declared dependency edge.
- `TaskEvidenceId`: stable identity for one task-evidence record or reference.
- `TaskDefinitionReference`: immutable canonical reference to a task definition.
- `TaskInstanceReference`: immutable canonical reference to a task instance.
- `TaskOwnerReference`: immutable reference to the accountable owner of a task.
- `TaskAssigneeReference`: immutable reference to the selected performer of a future execution.
- `TaskAuthorityReference`: immutable reference to authority or governing approval relevant to a task decision.
- `TaskWorkflowReference`: immutable binding from task concepts back to a workflow identity or workflow instance context.
- `TaskStepReference`: immutable binding from a task concept to a workflow step context.
- `TaskEvidenceReference`: immutable reference to task evidence.
- `TaskFailureReference`: stable reference to a task failure outcome or task failure record.

## Domain Model
- K7 SHOULD reuse existing K1 identifiers and validated text types whenever the existing canonical type already captures the required meaning.
- K7 SHOULD reuse `WorkflowId`, `WorkflowInstance`, `WorkflowDefinition`, `WorkflowStepReference`, `OwnershipPath`, `TransitionAuthorityReference`, and evidence-reference conventions from K2-K6 when they already express the needed upstream identity.
- K7 MAY introduce task-specific identity and reference types only where no frozen upstream type provides the required meaning.
- Canonical public identity MUST be expressed through `TaskDefinitionId`, `TaskInstanceId`, `TaskDependencyId`, or `TaskEvidenceId`.
- `TaskId` MUST NOT exist as a public type and MUST NOT exist as an alias of `TaskInstanceId`.
- Generic umbrella references MUST NOT be introduced when `TaskDefinitionReference` or `TaskInstanceReference` already conveys the required meaning.
- `TaskDependencyId` remains canonical because dependency audit, traceability, evidence association, dependency history, and deterministic edge reference all require stable edge identity.
- Task references MUST remain data-only and MUST NOT imply repository lookup, runtime resolution, or executor access.
- A task entity MUST be representable entirely through explicit validated inputs.

## Normative Requirements
- Every task entity MUST have stable canonical identity.
- Identity MUST remain immutable after creation.
- Task identifiers MUST support deterministic equality.
- Display names MUST NOT serve as canonical identity.
- Public APIs MUST use `TaskDefinitionId` or `TaskInstanceId` explicitly where task-definition and task-instance meaning differs.
- `TaskId` MUST NOT be introduced for naming convenience.
- References MUST NOT introduce runtime resolution.
- A task MUST be representable without persistence.
- A task MUST NOT contain executor handles, runtime resources, or transport bindings.
- Task models MUST NOT introduce serialization requirements unless an approved existing project convention later requires them.
- K7 MUST NOT introduce a new dependency without approved authority.
- K7 MUST remain inside `kernel-domain` unless approved architecture states otherwise.
- Foundation APIs MUST be additive to frozen K1-K6 APIs.
- Validation outcomes such as `Valid`, `Invalid`, `Allowed`, `Rejected`, and `NoOp` MAY be expressed through existing frozen domain patterns rather than a single generic result enum.

## Invariants
- Canonical identity is explicit.
- Canonical identity is immutable.
- References are side-effect free.
- Task identity is independent of display naming.
- Task-definition identity and task-instance identity are not interchangeable.
- Foundational task types remain infrastructure-neutral.

## Validation Rules
- Missing canonical identity MUST be rejected.
- Empty or malformed task-reference values MUST be rejected deterministically.
- Reuse of upstream references MUST preserve the upstream type semantics exactly.
- Ambiguous public identity models MUST be rejected.
- Unsupported or contradictory foundational references MUST be rejected deterministically.

## Rejection Conditions
- Missing stable task identity
- Attempted introduction of public `TaskId`
- Attempted use of display name as canonical identity
- Reference forms that require runtime resolution
- Embedded runtime handle, scheduler handle, or executor handle
- Embedded persistence handle or network transport handle

## Compatibility Requirements
- K7 MUST remain additive to K1-K6.
- K7 MUST reuse frozen upstream identifiers and references wherever possible.
- K7 MUST NOT redefine workflow, authorization, event, or runtime identities.
- K7 MUST NOT introduce generic task identity aliases that duplicate `TaskInstanceId`.
- K7 MUST preserve K6 public API compatibility.

## Public API Expectations
- K7 foundation SHOULD expose strongly typed identifiers and references with constructor validation and read-only accessors.
- K7 foundation SHOULD follow existing `new(...) -> DomainResult<Self>` patterns and immutable getter patterns used by K1-K6.
- K7 foundation MUST NOT expose mutable identity setters.

## Error Expectations
- K7 foundation SHOULD use task-specific domain errors only when no existing `DomainError` variant expresses the failure cleanly.
- Error text MUST be deterministic and stable.
- Error values MUST distinguish invalid task identity from invalid task binding.

## Test Requirements
- Stable identity construction
- No public `TaskId`
- Deterministic equality
- Invalid identity rejection
- Reference immutability
- No runtime resolution
- Clear separation of `TaskDefinitionId` and `TaskInstanceId`
- Compatibility with frozen K1-K6 reference types
- No infrastructure-bearing fields

## Purpose
Define the immutable description of a governed task type without representing a running or assigned task.

## Scope
- Immutable task-definition identity and version
- Task-definition inputs, outputs, requirements, evidence, completion, and failure-policy references
- Compatibility with K6 workflow definition and workflow step references

## Non-Goals
- Live task execution state
- Agent or runtime instance embedding
- Scheduler configuration
- Persistence schema
- Event publication infrastructure

## Terminology
- `TaskDefinition`: immutable description of a governed task type.
- `TaskDefinitionVersion`: stable version of a task definition.
- `TaskDefinitionName`: human-readable name that is not canonical identity.
- `TaskDescription`: validated description of task intent.
- `TaskKind`: canonical category of governed work.
- `TaskInputContract`: declared required inputs.
- `TaskOutputContract`: declared expected outputs where applicable.
- `TaskRequirement`: general requirement attached to a task definition.
- `TaskCapabilityRequirement`: required capability for future execution eligibility.
- `TaskEvidenceRequirement`: evidence required for readiness, completion, or failure.
- `TaskCompletionRequirement`: completion condition required before a task may become completed.
- `TaskFailurePolicyReference`: immutable reference to a declared failure policy.

## Domain Model
- A `TaskDefinition` describes governed work and MUST remain distinct from a `TaskInstance`.
- Definition identity and version SHOULD follow frozen K1 version and identifier conventions where reuse is possible.
- A task definition MAY bind to a `WorkflowDefinition` and `WorkflowStepReference` without modifying K6 workflow semantics.

## Normative Requirements
- A `TaskDefinition` describes work but does not represent a running task.
- Definition identity and version MUST be stable.
- A definition MUST declare required inputs.
- A definition MUST declare expected outputs where applicable.
- A definition MUST declare completion conditions.
- A definition MAY declare required evidence.
- A definition MAY declare capability requirements.
- A definition MUST NOT contain a live agent instance.
- A definition MUST NOT contain executor state.
- A definition MUST NOT mutate after publication or activation.
- Changes MUST create a new version or identity according to existing version conventions.
- Invalid or incomplete definitions MUST be rejected deterministically.
- Definitions MUST remain compatible with `WorkflowDefinition` and workflow step references without changing K6 semantics.

## Invariants
- Definition identity is immutable.
- Definition version is explicit.
- Completion expectations are explicit.
- Evidence requirements are explicit when present.
- Definitions remain side-effect free.

## Validation Rules
- Missing identity MUST be rejected.
- Missing required name MUST be rejected where naming is required by the public API.
- Invalid version MUST be rejected.
- Empty completion contract MUST be rejected.
- Duplicate requirements MUST be rejected.
- Contradictory input or output constraints MUST be rejected.
- Unsupported references MUST be rejected.
- Self-inconsistent evidence requirements MUST be rejected.

## Rejection Conditions
- Missing task-definition identity
- Missing required task-definition name
- Invalid task-definition version
- Empty completion requirement set
- Duplicate requirement declarations
- Contradictory input or output contracts
- Unsupported workflow or step binding
- Evidence requirement contradiction

## Compatibility Requirements
- K7 task definitions MUST remain additive to K6 workflow definitions.
- K7 MUST NOT redefine workflow step semantics.
- K7 MUST reuse frozen version conventions and validated reference patterns.

## Public API Expectations
- `TaskDefinition` SHOULD follow immutable constructor-plus-getter conventions used by `WorkflowDefinition`.
- Definitions SHOULD expose read-only accessors for identity, version, name, requirements, workflow bindings, evidence requirements, and completion requirements.

## Error Expectations
- Definition-construction errors MUST distinguish identity failure, version failure, contract incompleteness, and duplicate requirements.
- Errors MUST be deterministic and testable.

## Test Requirements
- Valid construction
- Identity and version preservation
- Input and output contract preservation
- Completion requirement preservation
- Evidence requirement preservation
- Duplicate rejection
- Contradictory constraint rejection
- Workflow compatibility preservation

- No execution state or runtime handle is introduced.

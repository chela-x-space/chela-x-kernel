# 1. Purpose

This specification defines the Canonical Event Envelope used throughout
the CHELA-X Enterprise Event System.

Every Enterprise Event shall be represented by exactly one canonical
Event Envelope.

The Event Envelope establishes the common structure for all Enterprise
Events regardless of origin.

---

# 2. Objectives

The Canonical Event Envelope provides:

- Stable identity
- Immutable structure
- Version compatibility
- Enterprise traceability
- Correlation
- Causation
- Classification
- Validation
- Replay support

The Event Envelope becomes the contractual boundary of the Enterprise
Event System.

---

# 3. Scope

This specification defines:

- Event metadata
- Required fields
- Optional fields
- Validation requirements
- Versioning rules

This specification does not define:

- Event categories
- Event streams
- Event replay
- Event storage
- Event transport

Those subjects belong to later K5 milestones.

- Deterministic
- Versioned
- Traceable

---

# 6. Canonical Event Envelope

Every Enterprise Event shall contain exactly one Event Envelope.

The Event Envelope is the immutable metadata surrounding the Event
Payload.

The Event Payload may differ between Event types.

The Event Envelope remains structurally consistent across the entire
enterprise.

---

# 7. Required Fields

Every Event Envelope shall contain the following mandatory fields.

| Field | Required | Description |
|--------|----------|-------------|
| event_id | YES | Canonical Event Identifier |
| event_type | YES | Enterprise Event Type |
| event_version | YES | Event schema version |
| occurred_at | YES | Event occurrence timestamp |
| recorded_at | YES | Event recording timestamp |
| source | YES | Originating component |
| subject | YES | Enterprise object affected |
| payload | YES | Event data |
| classification | YES | Security classification |
| trace | YES | Traceability information |

An Event missing any mandatory field shall be rejected.

---

# 8. Optional Fields

The following fields are optional.

| Field | Purpose |
|--------|----------|
| correlation_id | Workflow correlation |
| causation_id | Parent event |
| tenant_id | Multi-tenant support |
| tags | Search metadata |
| labels | Human-readable grouping |
| attachments | External references |
| evidence | Supporting evidence |
| checksum | Integrity verification |

Optional fields shall never alter canonical Event semantics.

---

# 9. Canonical Structure

Conceptually every Event shall follow this structure.

```text
Event Envelope

├── Identity
├── Metadata
├── Classification
├── Traceability
├── Payload
└── Integrity
```

Payload contents differ by Event type.

Envelope structure remains constant.

---

# 10. Identity

Event Identity consists of:

- Event ID
- Event Version

Event Identity never changes.

Identity shall remain valid during:

- storage
- replication
- synchronization
- serialization
- replay

Identity shall never be regenerated.

---

# 11. Source

Every Event identifies its origin.

Typical sources include:

- Runtime
- Workflow
- Task
- Execution
- API Gateway
- Studio
- Memory
- Security

Source identifies where the accepted Event originated.

It does not identify transport technology.

---

# 12. Subject

The Subject identifies the Enterprise Entity affected by the Event.

Examples include:

- Runtime
- Workflow
- Task
- Agent
- User
- Organization
- Project
- Memory Record

Every Event has exactly one canonical Subject.

---

# 13. Payload

Payload contains business information.

The Payload is governed by Event Type.

Payload shall never redefine Event metadata.

Metadata and Payload remain independent.

---

# 14. Timestamp

Every Event contains:

occurred_at

The time the enterprise fact occurred.

recorded_at

The time the Event entered the Enterprise Event System.

These timestamps may differ.

Both timestamps are immutable.

---

# 15. Classification

Every Event possesses one security classification.

Classification follows the CHELA-X CES Security Classification model.

Supported classifications include:

- PUBLIC
- INTERNAL
- CONFIDENTIAL
- RESTRICTED
- CRITICAL

Classification governs visibility,
handling,
distribution,
and retention.

Classification never changes Event meaning.

---

# 16. Integrity

Every Event Envelope shall preserve integrity.

Integrity guarantees:

- Envelope completeness
- Immutable identity
- Consistent metadata
- Stable payload boundaries
- Traceability
- Version compatibility

Integrity validation occurs before an Event is accepted.

---

# 17. Envelope Invariants

The following invariants always apply:

- Exactly one Event Identity
- Exactly one Event Type
- Exactly one Event Version
- Exactly one Payload
- Immutable metadata
- Immutable timestamps
- Immutable classification
- Stable serialization

Violation of any invariant results in Event rejection.

---

# 18. Canonical Determination

The Canonical Event Envelope defines the mandatory structure of every
Enterprise Event within the CHELA-X Kernel.

All subsequent K5 specifications shall extend this Envelope without
changing its fundamental structure.

This specification establishes the canonical Event Envelope baseline for
the Enterprise Event System.

---

# 19. Correlation

Correlation groups Events that participate in the same enterprise
activity, workflow, transaction, incident, or operational context.

The `correlation_id` field is optional at the Envelope level.

When present, it shall identify one canonical correlation context.

Multiple Events may share the same Correlation Identifier.

Correlation indicates relationship.

Correlation does not prove that one Event caused another Event.

The Correlation Identifier shall:

- remain stable for the complete correlation context
- be transport independent
- be serialization independent
- never replace Event Identity
- never replace Workflow Identity
- never replace Task Identity

An Event may exist without a Correlation Identifier when no governed
correlation context exists.

---

# 20. Causation

Causation identifies the accepted Event that directly caused another
Event.

The `causation_id` field references a canonical Event Identifier.

When Event B is directly caused by Event A:

    Event A
        │
        ▼
    Event B

Event B records the Event Identifier of Event A as its `causation_id`.

Causation shall not reference:

- a Command Identifier
- a Task Identifier
- a Workflow Identifier
- an Execution Session Identifier
- an unknown Event
- the Event itself

A root Event may omit `causation_id`.

Causation lineage shall remain immutable after Event acceptance.

---

# 21. Traceability

Every Event shall preserve sufficient traceability to explain:

- what occurred
- when it occurred
- where it originated
- which enterprise object was affected
- which authority context applied
- which evidence supports the fact
- which earlier Event directly caused it
- which broader activity correlates it

Traceability shall be explicit.

Traceability shall not depend upon log text, storage location, process
memory, or external interpretation.

The canonical trace structure shall support references to:

- Actor Identity
- Runtime Identity
- Workflow Identity
- Task Identity
- Execution Session Identity
- Decision Identity
- Evidence Identity
- Correlation Identity
- Causation Event Identity

Only references applicable to the Event are populated.

---

# 22. Validation Rules

An Event Envelope shall be rejected when any mandatory validation rule
fails.

Mandatory validation includes:

1. `event_id` is present and structurally valid.
2. `event_type` is present and canonical.
3. `event_version` is supported.
4. `occurred_at` is present and valid.
5. `recorded_at` is present and valid.
6. `recorded_at` does not precede `occurred_at` unless an approved
   ingestion rule explicitly permits historical recording.
7. `source` is present and valid.
8. `subject` is present and valid.
9. `payload` conforms to the Event Type schema.
10. `classification` is a supported CES classification.
11. `trace` is structurally valid.
12. `causation_id` does not equal `event_id`.
13. Referenced identities use the correct identity type.
14. Unknown mandatory fields are rejected when strict schema validation
    applies.
15. Event acceptance does not mutate an existing Event.

Validation shall occur before recording.

Validation failure shall produce a deterministic canonical error.

Rejected Event candidates shall not enter canonical enterprise history.

---

# 23. Validation Precedence

Validation precedence shall remain deterministic.

The canonical precedence is:

1. Envelope structure
2. Mandatory field presence
3. Identity validity
4. Version compatibility
5. Timestamp validity
6. Source validity
7. Subject validity
8. Classification validity
9. Trace validity
10. Correlation and causation validity
11. Payload schema validity
12. Integrity validation
13. Duplicate Event detection

A failure detected at an earlier validation stage shall not be masked by a
later validation failure.

Equivalent invalid inputs shall produce equivalent canonical outcomes.

domain layer.

The implementation shall not introduce:

- a database
- an Event Store
- a message broker
- asynchronous delivery
- network transport
- event publishing
- event subscribers
- replay execution
- background workers

The implementation should define:

- Event Identifier
- Event Type
- Event Version
- Event Source
- Event Subject
- Event Classification
- Event Trace
- Event Envelope
- Canonical validation errors
- Deterministic constructors

# Appendix A — Canonical Identity Conventions

This appendix defines the canonical identifier conventions used by the
Enterprise Event System.

Identifier formats are normative.

Implementations shall preserve these semantic meanings regardless of the
underlying identifier generation strategy.

| Identity Type | Canonical Format | Example |
|---|---|---|
| EventId | `CX-EVT-` followed by 6 ASCII digits | `CX-EVT-000001` |
| CorrelationId | `CX-COR-` followed by 6 ASCII digits | `CX-COR-000001` |

`causation_id` does not define a separate identity type.

A causation reference shall contain the `EventId` of the directly
causing parent Event.

Example:

    event_id = CX-EVT-000002
    causation_id = CX-EVT-000001

The following rules are mandatory:

- Event Identity uses `EventId`.
- Correlation Context uses `CorrelationId`.
- Causation references use `EventId`.
- Identifiers are supplied by callers.
- The Domain layer does not generate identifiers.
- Identifier values remain immutable after Event acceptance.
- Identity format is independent of storage and transport.

Identifier prefixes improve readability and semantic validation.

They do not replace global uniqueness requirements.

# K7.5 Task Priority And Readiness

## Status
ARCHITECTURE APPROVED

## Milestone
K7 Task Engine

## Architecture Authority
- `docs/kernel-architecture/08-task-architecture.md`
- `docs/kernel-architecture/01-kernel-overview.md`
- `docs/kernel-architecture/16-traceability.md`

## Purpose
Define deterministic priority and side-effect-free readiness evaluation for tasks without introducing scheduling or execution.

## Scope
- Task-priority vocabulary
- Readiness evaluation
- Readiness blockers and evidence
- Readiness interaction with ownership, assignment, dependencies, authorization, and evidence prerequisites

## Non-Goals
- Scheduling
- Wall-clock ordering
- Capacity reservation
- Worker dispatch
- Execution

## Terminology
- `TaskPriority`: explicit priority attached to a task.
- `TaskPriorityClass`: stable named priority class.
- `TaskPriorityValue`: comparable priority value.
- `TaskReadiness`: derived domain evaluation for future execution eligibility.
- `TaskReadinessDecision`: deterministic readiness outcome with canonical values `Ready` or `Blocked`.
- `TaskReadinessRequirement`: one condition that must hold for readiness.
- `TaskReadinessBlocker`: stable blocker preventing readiness.
- `TaskReadinessEvidence`: evidence used in readiness evaluation.

## Domain Model
- Priority is governance metadata.
- Readiness is a pure evaluation over explicit task facts.
- `Ready` and `Blocked` are readiness outcomes, not lifecycle states.
- Readiness is distinct from assignment and distinct from execution.

## Normative Requirements
- Priority MUST be explicit where used.
- Priority MUST NOT itself schedule or execute work.
- Priority ordering MUST be deterministic.
- Equal priority MUST remain a valid result.
- K7 MUST NOT invent wall-clock scheduling.
- Readiness evaluation MUST be side-effect free.
- Readiness MUST distinguish ready from blocked.
- Readiness evaluation SHOULD return stable blocker reasons.
- A task MUST NOT be ready when required input is missing.
- A task MUST NOT be ready when ownership requirements are unmet.
- A task MUST NOT be ready when assignment is required but absent.
- A task MUST NOT be ready when dependencies are incomplete.
- A task MUST NOT be ready when required authorization is denied.
- A task MUST NOT be ready when the task is terminal.
- A task MUST NOT be ready when required evidence prerequisites are missing.
- A task MAY be ready without an assignee only if the definition and policy explicitly permit later assignment.
- Readiness MUST NOT reserve resources.
- Readiness MUST NOT claim agent capacity.
- Readiness MUST NOT change assignment.
- Readiness MUST NOT change task lifecycle.
- Readiness MUST NOT mutate workflow state.

## Invariants
- Priority is not scheduling.
- Readiness is not execution.
- Readiness is not lifecycle state.
- Blocker categories are stable and deterministic.
- Readiness evaluation does not mutate task, workflow, or runtime state.

## Validation Rules
- Unsupported priority class MUST be rejected.
- Contradictory readiness requirements MUST be rejected.
- Terminal tasks MUST be not-ready.
- Missing blocker evidence MUST produce deterministic blocked outcomes where blocker evidence is required.

## Rejection Conditions
- Invalid priority value
- Unsupported priority class
- Self-contradictory readiness requirement
- Attempt to interpret readiness as scheduling authorization

## Compatibility Requirements
- Readiness MUST consume K3 and K4 facts without redefining them.
- Readiness MUST consume K6 dependency and workflow context without mutating workflow state.
- K7 MUST remain compatible with frozen K2 transition and sequence conventions if readiness outcomes later reuse them.

## Public API Expectations
- K7 SHOULD expose explicit task-priority and readiness types rather than raw integers or booleans alone.
- Readiness APIs SHOULD return `TaskInstanceId`, `Ready` or `Blocked`, stable blocker categories, and the validated facts or references used in the evaluation.

## Error Expectations
- Invalid priority construction MUST be distinct from blocked readiness outcomes.
- Readiness blockers MUST be domain outcomes, not infrastructure errors.

## Test Requirements
- Priority ordering determinism
- Equal-priority stability
- `Ready` and `Blocked` remain derived outcomes
- Missing-input blocker
- Missing-owner blocker
- Missing-assignment blocker where required
- Dependency blocker
- Authorization-denied blocker
- Terminal-state blocker
- Readiness does not mutate lifecycle or assignment
- No mutation and no scheduling side effect

## Traceability
- `docs/kernel-architecture/08-task-architecture.md`: priority, dependencies, completion, failure
- `docs/kernel-architecture/01-kernel-overview.md`: task governance placement
- CES TRACEABILITY: PENDING AUTHORITATIVE MAPPING

## Acceptance Criteria
- Priority and readiness are explicitly separated from execution.
- Readiness decisions are deterministic and testable.
- No scheduler semantics are introduced.

## Deferred Concerns
- Exact canonical priority classes
- Advanced prioritization policies
- Capacity-sensitive readiness beyond frozen upstream facts

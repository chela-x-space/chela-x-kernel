# K6.4 Workflow Transition Control

## Status
Draft

## Purpose
Define deterministic workflow transition control over workflow definitions and instances.

## Scope
- Workflow lifecycle states and legal transitions.
- Transition guards for approval, start, resume, failure, cancellation, recovery, and archival.
- Failure-code and evidence prerequisites already reflected by current kernel-domain workflow transition validation.

## Out of Scope
- Policy evaluation logic itself.
- Task state transitions.
- Execution outcomes, schedulers, or timers.

## Canonical Terminology
- Transition Request: a requested move from one workflow state to another.
- Workflow Guard: the upstream prerequisite set that must validate before a transition may be accepted.
- Transition Outcome: the deterministic result of workflow transition validation.

## Structures
- `WorkflowState`.
- `WorkflowTransitionRequest`.
- `WorkflowLifecycleGuards`.
- `WorkflowTransitionOutcome`.
- `validate_workflow_transition`.

## Invariants
- Only the approved workflow state map is legal.
- Archived workflows are terminal and non-executable.
- Start or resume cannot bypass policy, authorization, delegation, decision, scope, participants, or audit evidence.
- Failed-to-ready recovery requires bounded retry and fresh revalidation.
- Failure transitions require stable failure codes.

## Validation Precedence
1. No-op and terminal-state short-circuit checks.
2. Allowed state-map validation.
3. Approval guard: valid upstream decision.
4. Start or resume guards: policy, authorization, delegation, decision, scope, participants, upstream outcomes, and audit evidence.
5. Recovery guards: bounded retry and fresh revalidation.
6. Failure guards: stable failure code.

## Deterministic Outcomes
- Identical requests and guards return identical transition outcomes.
- Guard failures reject before later transition effects are considered.
- Transition validation remains pure and independent of infrastructure state.

## K1-K5 Relationships
- K2 is the direct implementation authority for workflow transition control.
- K3 supplies the consumed authorization, delegation, and decision prerequisites.
- K5 supplies the event facts that may be emitted only after transition acceptance.

## CES Traceability
- `CES-B0-030.9`: allowed workflow states and transitions.
- `CES-B0-030.10`: command intent to state transition linkage.
- `CES-B0-030.13`: stable workflow failure codes.
- `CES-B0-030.14`: bounded retry and recovery revalidation.
- `CES-B0-030.17`: audit evidence for decision-relevant transitions.

## API Implications
- K6 transition-control APIs should extend `state.rs` rather than invent a parallel workflow transition engine.
- Future K6 validation functions must preserve existing transition semantics and guard ordering.
- No transition API may mutate state outside explicit outcome records.

## Acceptance Criteria
- Transition control stays aligned with the existing `WorkflowState` map and `validate_workflow_transition` semantics.
- Validation ordering is explicit, deterministic, and evidence-bound.
- K6 transition control does not absorb task or execution lifecycle semantics.

## Final Determination
K6 transition control is the pure-validation boundary that converts workflow state intent into deterministic accepted or rejected outcomes under approved CES guards.

# K6.6 Workflow Authorization And Policy

## Status
Draft

## Purpose
Define how K6 workflow progression consumes approved policy, authorization, delegation, and SoD outcomes.

## Scope
- Upstream policy, authorization, delegation, decision, scope, participant, and SoD prerequisites for workflow approval, start, resume, and recovery.
- Explicit deny and indeterminate blocking behavior.

## Out of Scope
- Policy authoring.
- Authorization evaluation engines.
- Delegation lifecycle redesign.
- Independent workflow authority or exception systems.

## Canonical Terminology
- Upstream Policy Result: the consumed policy outcome governing workflow safety.
- Consumed Authorization Result: the governing authorization outcome attached to one workflow action.
- Consumed Delegation Result: the bounded delegation evidence permitting one workflow role or action.

## Structures
- Workflow guards in `WorkflowLifecycleGuards`.
- Audit-linked workflow references in `WorkflowAuditEvidenceReference`.
- Existing K3 authorization, decision, and delegation records consumed by workflow.

## Invariants
- Workflow follows Policy -> Authorization -> Delegation -> Decision dependency direction.
- Explicit deny blocks workflow execution.
- Participant assignments never widen scope or bypass SoD.
- Workflow does not create or reinterpret authority.

## Validation Precedence
1. Tenant and scope boundary compatibility.
2. Security classification compatibility.
3. Explicit deny and mandatory policy outcomes.
4. Authorization validity.
5. Delegation validity and SoD independence.
6. Decision validity.
7. Participant and upstream-outcome allowance.

## Deterministic Outcomes
- Identical policy, authorization, delegation, and decision evidence yields identical workflow-guard outcomes.
- `Indeterminate` blocks approval and execution.
- Deny-class failures reject before task or event-side considerations.

## K1-K5 Relationships
- K3 is the direct authority layer consumed by K6.
- K2 transition control already enforces workflow guard requirements for approved transitions.
- K4 constrains agent participation through trust and supervision compatibility.
- K5 later records accepted workflow outcomes as immutable events.

## CES Traceability
- `CES-B0-028.9`: deterministic policy evaluation order and explicit deny precedence.
- `CES-B0-029.15`: separation-of-duties independence requirements.
- `CES-B0-029.18`: append-only delegation audit evidence.
- `CES-B0-030.5`: workflow participant-role constraints.
- `CES-B0-030.12`: workflow validation short-circuit behavior.

## API Implications
- K6 should consume K3 outcomes by reference, not duplicate K3 evaluators.
- Future workflow-authorization APIs should be pure and evidence-based.
- No K6 API may hide deny, indeterminate, or SoD outcomes.

## Acceptance Criteria
- Workflow authorization remains wholly downstream from K3.
- Validation ordering preserves explicit deny and SoD precedence.
- K6 documentation introduces no independent workflow permission system.

## Final Determination
K6 workflow authorization and policy is a consumption boundary over K3 evidence, not a new authority system, and it must short-circuit deterministically on deny, indeterminate, and SoD failures.

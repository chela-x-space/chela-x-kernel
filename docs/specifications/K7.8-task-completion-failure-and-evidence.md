## Purpose
Define explicit domain contracts for task completion, failure, and evidence without introducing execution or recovery infrastructure.

## Scope
- Completion
- Completion outputs
- Failure representation
- Evidence identity and metadata
- Distinction between domain rejection, declared failure, dependency blockage, and future execution failure

## Non-Goals
- Execution
- Recovery execution
- Retry scheduling
- Evidence storage payloads
- Persistence

## Terminology
- `TaskCompletion`: accepted completion fact for a task.
- `TaskCompletionOutcome`: completion result classification.
- `TaskCompletionResult`: immutable result value for completion.
- `TaskOutput`: declared task output.
- `TaskOutputReference`: immutable reference to task output.
- `TaskFailure`: accepted task failure fact.
- `TaskFailureCode`: stable machine-comparable failure identifier or category.
- `TaskFailureCategory`: stable higher-level failure class.
- `TaskFailureReason`: human-readable supporting context.
- `TaskFailureReference`: stable reference to a failure outcome or failure record.
- `TaskFailureEvidence`: evidence supporting failure.
- `TaskEvidence`: evidence attached to a task.
- `TaskEvidenceType`: canonical type of evidence.
- `TaskEvidenceMetadata`: infrastructure-neutral evidence metadata.
- `TaskEvidenceValidation`: deterministic evidence-validation outcome.
- `TaskRecoveryReference`: reference to a future recovery policy or path without executing it.

## Domain Model
- Completion, failure, and evidence are distinct governed concepts.
- Completion is not inferred from informal reports.
- Failure is not equivalent to domain validation rejection.
- `TaskFailureCode` is canonical machine-comparable identity for failure meaning.
- `TaskFailureCategory` is deterministic stable classification.
- `TaskFailureReason` is supporting human-readable context only and is not canonical identity.
- `TaskFailureReference` is traceable reference and MUST NOT replace failure code.
- Evidence references MUST remain identity-bearing and infrastructure-neutral.

## Normative Requirements
- Completion MUST be explicit.
- Completion MUST validate required outputs.
- Completion MUST validate required evidence.
- Completion MUST reject missing mandatory completion conditions.
- Completion MUST NOT be inferred only from an assignee report.
- Failure MUST include a stable machine-comparable code.
- Failure MUST include deterministic category.
- Failure SHOULD include human-readable context without using it as canonical identity.
- Failure MUST distinguish domain rejection from execution failure.
- Authorization denial MUST remain distinct from task failure.
- Dependency blockage MUST remain distinct from task failure.
- K7 MUST model execution failure results but MUST NOT execute or recover work.
- Recovery execution remains deferred.
- Retry scheduling remains deferred.
- Evidence MUST have stable identity.
- Evidence MUST identify its subject task.
- Evidence MUST identify its producer or authority where required.
- Evidence references MUST not require embedded binary payloads.
- Evidence metadata MUST remain domain-safe and infrastructure-neutral.
- Duplicate evidence identity MUST be rejected where detectable.
- Completion after terminal failure MUST be rejected unless an approved recovery transition exists.
- Completed task output MUST not be silently mutable.
- Archived evidence MUST remain referentially traceable according to the architecture, without implementing storage.

## Invariants
- Completion is explicit.
- Failure is explicit.
- Evidence identity is stable.
- Evidence metadata is infrastructure-neutral.
- Terminal completion outputs are immutable after acceptance.

## Validation Rules
- Missing mandatory completion evidence MUST be rejected.
- Missing mandatory outputs MUST be rejected.
- Missing stable failure code MUST be rejected.
- Missing deterministic failure category MUST be rejected.
- Missing subject task binding for evidence MUST be rejected.
- Duplicate evidence identity MUST be rejected where detectable.
- Contradictory completion and failure facts MUST be rejected.

## Rejection Conditions
- Missing completion requirements
- Missing mandatory outputs
- Missing mandatory evidence
- Missing stable failure code
- Missing deterministic failure category
- Duplicate task evidence identity
- Completion after terminal failure without approved recovery path
- Evidence metadata that embeds infrastructure handles or binary payload requirements

## Compatibility Requirements
- K7 evidence references SHOULD follow the stable-reference and audit patterns already used in K3, K5, and K6.
- K7 MUST NOT reinterpret K6 workflow failure or recovery semantics.
- K7 MUST remain compatible with future K5 event construction by reference only, without introducing publication semantics.

## Public API Expectations
- K7 SHOULD expose separate types for completion, failure, and evidence.
- Completion and failure results SHOULD be immutable.
- Evidence types SHOULD expose identity, subject, authority or producer, and validated metadata through read-only accessors.

## Error Expectations
- Domain validation failure, task-declared failure, future execution failure, authorization denial, and dependency blockage MUST remain distinguishable.
- Error or outcome text MUST be stable and deterministic.

## Test Requirements
- Explicit completion construction
- Missing-output rejection
- Missing-evidence rejection
- Stable failure-code preservation
- Failure category preservation
- Failure reason remains non-canonical
- Execution failure distinct from domain rejection
- Duplicate-evidence rejection
- Evidence subject preservation
- No execution or recovery side effect
- Distinction between rejection and declared failure

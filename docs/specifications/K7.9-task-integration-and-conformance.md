## Purpose
Define the K7 milestone closure and cross-layer conformance obligations for Task Engine implementation. This document is a closure and compatibility specification, not the K7 milestone name.

## Scope
- K1-K6 compatibility obligations
- Validation and engineering-gate obligations
- Task-engine integration boundaries
- Documentation closure and API-freeze expectations

## Non-Goals
- Event-bus creation
- Implicit event publication
- Persistence
- Async runtime
- Network dependencies
- Workflow side effects
- Task execution

## Terminology
- `Task Integration`: additive use of frozen upstream contracts by K7.
- `Task Conformance`: proof that K7 remains compatible with frozen architecture, APIs, and validation obligations.
- `Compatibility Test`: deterministic test proving upstream APIs remain usable.
- `Conformance Matrix`: traceable mapping from K7 work items to frozen upstream obligations and validation gates.

## Domain Model
- K7 remains `Task Engine`.
- `Task Integration and Conformance` is a K7 closure slice, not the milestone name.
- K7 sits between Workflow and future Execution:

```text
Workflow Engine
    ↓
Task Engine
    ↓
Future Execution Engine
```

## Normative Requirements
- K7 MUST not modify frozen upstream APIs.
- K7 additions MUST be backward compatible.
- Existing K1-K6 tests MUST continue to pass.
- K7 MUST include compatibility tests proving frozen APIs remain usable.
- K7 MUST include cross-module domain tests where approved.
- K7 MUST not create an event bus.
- K7 MUST not publish events implicitly.
- K7 MAY define pure event-construction data only if explicitly required and compatible with K5 patterns.
- K7 MUST not create workflow side effects.
- K7 MUST not execute tasks.
- K7 MUST not add persistence.
- K7 MUST not introduce async.
- K7 MUST not introduce network dependencies.
- K7 MUST pass all engineering gates.
- K7 public API MUST be reviewed before freeze.
- API freeze MUST occur only after K7 implementation and host validation are complete.
- K7 documentation closure MUST update API, traceability, validation, implementation plan, backlog, README, and CHANGELOG.

## Invariants
- K7 is additive.
- K6 workflow API remains frozen and unchanged.
- K7 readiness is not execution.
- K7 compatibility proof is mandatory before milestone acceptance.

## Validation Rules
- All K1-K6 compatibility suites MUST continue to pass.
- K7 compatibility suites MUST prove that frozen APIs remain usable.
- Engineering gates MUST include formatting, build, tests, lint, docs, doc tests, and clean diff checks.
- Host validation remains the authoritative final validation source when available.

## Rejection Conditions
- Any K7 implementation that changes frozen K1-K6 semantics
- Any K7 implementation that introduces runtime infrastructure
- Any K7 implementation that creates implicit event publication or workflow side effects
- Any K7 milestone closure attempt without compatibility evidence

## Compatibility Requirements
- K1: identifiers, references, and value primitives remain reusable
- K2: state snapshots, sequence, transition outcomes, and lifecycle conventions remain reusable
- K3: authorization requests and decisions remain consumable by reference
- K4.1/K4.2: runtime registration, eligibility, lifecycle, and supervision facts remain consumable by reference
- K5: event-envelope and event vocabulary compatibility remain reusable without publication infrastructure
- K6: workflow definition, workflow instance, workflow step, authorization, event integration, and failure or recovery APIs remain unchanged and usable

## Public API Expectations
- K7 public APIs SHOULD be additive within `kernel-domain`.
- K7 public APIs SHOULD follow frozen constructor and accessor conventions used by K1-K6.
- K7 API freeze MUST occur only after host-verified milestone completion.

## Error Expectations
- Conformance failures MUST distinguish compatibility breakage from local validation failures.
- K7 MUST NOT collapse cross-layer domain failures into a single undifferentiated error.

## Test Requirements
- Compatibility tests for K1, K2, K3, K5, and K6 surfaces used by K7
- Cross-module domain tests where approved
- Assignment-versus-lifecycle separation tests
- Readiness-versus-lifecycle separation tests
- Failure-versus-rejection separation tests
- Dependency satisfaction versus lifecycle-mutation tests
- No-side-effect tests proving no event publication, no persistence, no execution, and no workflow mutation
- Engineering-gate validation evidence

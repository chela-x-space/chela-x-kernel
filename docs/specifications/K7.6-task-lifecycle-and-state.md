# K7.6 Task Lifecycle And State

## Status
ARCHITECTURE APPROVED

## Milestone
K7 Task Engine

## Architecture Authority
- `docs/kernel-architecture/08-task-architecture.md`
- `docs/kernel-architecture/01-kernel-overview.md`
- `docs/kernel-architecture/16-traceability.md`
- `crates/kernel-domain/src/state.rs`

## Purpose
Define deterministic task lifecycle vocabulary and pure transition validation compatible with frozen K2 state patterns.

## Scope
- Canonical task states
- Allowed and rejected transition edges
- No-op behavior
- Authority, reason, evidence, and sequence expectations

## Non-Goals
- Workflow-state redesign
- Implicit persistence
- Implicit event emission
- Execution of task work

## Terminology
- `TaskState`: canonical task lifecycle state.
- `TaskStateSnapshot`: immutable task-state snapshot.
- `TaskTransitionRequest`: requested state change for a task.
- `TaskTransitionDecision`: deterministic task transition outcome.
- `TaskTransitionControl`: pure validator for task transitions.

## Domain Model
- K7 SHOULD reuse existing K2 sequence, snapshot, and transition-outcome patterns where practical.
- Canonical task lifecycle states are `Pending`, `InProgress`, `Completed`, `Failed`, `Cancelled`, and `Archived`.
- `Pending` is the initial task-instance state.
- Operational terminal states are `Completed`, `Failed`, and `Cancelled`.
- `Archived` is the record terminal state.
- `Assigned` belongs only to `TaskAssignmentStatus`.
- `Ready` is only `TaskReadinessDecision`.
- `Blocked` is only readiness outcome with `TaskReadinessBlocker`.
- `Draft` remains deferred concern and is not `TaskInstance` state in K7.

## Normative Requirements
- State transitions MUST be explicit.
- State transitions MUST be deterministic.
- Same-state requests MUST return deterministic no-op outcomes.
- Illegal edges MUST be rejected.
- Terminal states MUST reject operational transitions.
- Completion MUST require completion validation.
- Failure MUST require a stable failure code and deterministic failure category.
- Cancellation MUST require authority and reason.
- Archival MUST require valid archival authority.
- Archival MUST not be equivalent to deletion.
- Transition validation MUST be pure domain behavior.
- Transition control MUST not persist or emit events implicitly.
- Existing K2 state and transition primitives SHOULD be reused.
- Sequence mismatch MUST return `RejectedTransition` with reason `SequenceMismatch`.
- Allowed transitions are only:
  - `Pending -> InProgress`
  - `Pending -> Cancelled`
  - `InProgress -> Completed`
  - `InProgress -> Failed`
  - `InProgress -> Cancelled`
  - `Completed -> Archived`
  - `Failed -> Archived`
  - `Cancelled -> Archived`
- K7 MUST not change `WorkflowState` semantics.

## Invariants
- State snapshots are explicit.
- Sequence values are explicit where sequencing is used.
- Terminal states remain terminal.
- Archival preserves history.
- Readiness and assignment are separate from lifecycle.

## Validation Rules
- Initial state MUST be explicit.
- Initial state MUST be `Pending`.
- No-op requests MUST return deterministic no-op outcomes.
- No-op MUST NOT change state, sequence, ownership, assignment, or evidence.
- Illegal transition edges MUST return deterministic rejection outcomes.
- Missing required authority, reason, or evidence MUST be rejected.
- Sequence mismatch MUST reject without changing state, sequence, ownership, assignment, evidence, or any external side effect.
- `Pending -> InProgress` MUST require readiness `Ready`, required assignment if required by definition, allowed authorization, satisfied dependencies, and matching sequence.
- `Pending` or `InProgress -> Cancelled` MUST require valid authority, non-empty reason, and matching sequence.
- `InProgress -> Completed` MUST require completion conditions, required outputs, required evidence, and matching sequence.
- `InProgress -> Failed` MUST require stable failure code, deterministic failure category, required failure evidence if defined, and matching sequence.
- `Completed`, `Failed`, or `Cancelled -> Archived` MUST require valid archival authority and matching sequence.

## Rejection Conditions
- Unsupported state value
- Illegal transition edge
- Attempt to use `Draft`, `Ready`, `Assigned`, or `Blocked` as lifecycle state
- Missing authority for cancellation or other governed transition
- Missing stable failure code or deterministic failure category for failure transition
- Sequence mismatch
- Operational transition from terminal state

## Compatibility Requirements
- K7 lifecycle control MUST follow K2 transition-control patterns and error behavior where practical.
- K7 MUST NOT modify K2 workflow-state, delegation-state, or runtime-state semantics.
- K7 MUST reuse K2 sequence and transition-outcome conventions for no-op and rejected transitions where practical.
- Task-state control MUST remain additive in `kernel-domain`.

## Public API Expectations
- K7 SHOULD expose explicit task-state enums, state snapshots, transition requests, lifecycle guards if needed, and transition outcomes analogous to existing K2 patterns.
- Transition validators SHOULD remain pure and side-effect free.

## Error Expectations
- Invalid state construction MUST be distinct from transition rejection.
- Rejection outcomes MUST distinguish illegal transition, missing evidence, guard failure, and no-op where applicable.

## Test Requirements
- Canonical state construction
- Initial state is `Pending`
- Legal transition acceptance
- `Pending -> Completed` rejected
- `Pending -> Failed` rejected
- `InProgress -> Pending` rejected
- Illegal transition rejection
- Terminal-state protection
- Required authority, reason, evidence validation
- No-op behavior
- Sequence mismatch rejection
- Deterministic sequence behavior
- Workflow-state compatibility preservation

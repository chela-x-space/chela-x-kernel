# K6.3 Workflow Instance

## Status
Draft

## Purpose
Define the governed workflow-instance model derived from one approved workflow definition version.

## Scope
- Instance identity, definition-version linkage, ownership path, current state, context, and history inputs.
- Instance dependence on consumed policy, authorization, delegation, decision, and audit evidence.

## Out of Scope
- Task records or execution records.
- Database-backed history storage.
- Mutable side effects or orchestration workers.

## Canonical Terminology
- Workflow Instance: one execution of one approved workflow definition version.
- Current State: the active lifecycle state for one instance.
- Consumed Evidence Set: the policy, authorization, delegation, decision, and audit references required to justify one instance outcome.

## Structures
- `WorkflowStateSnapshot` is the current additive instance-state authority in `kernel-domain`.
- `StateSequence` preserves deterministic ordering of workflow state snapshots.
- `WorkflowAuditEvidenceReference` captures consumed upstream references for decision-relevant transitions.

## Invariants
- Every instance references exactly one approved definition version.
- Every instance belongs to one ownership path and one bounded scope lineage.
- Every instance state is explicit and sequence-bound.
- Instances preserve identity while lifecycle state evolves through validated transitions only.

## Validation Precedence
1. Workflow identity and subject ownership path.
2. Approved definition-version linkage.
3. Current lifecycle state and sequence validity.
4. Consumed upstream evidence completeness.
5. Audit-evidence presence for decision-relevant movement.

## Deterministic Outcomes
- Equivalent snapshots and evidence produce equivalent instance-validation outcomes.
- Missing or stale upstream evidence blocks instance advancement.
- Instance meaning remains reproducible from snapshot, sequence, and consumed evidence.

## K1-K5 Relationships
- K1 supplies `WorkflowId`, ownership path semantics, and stable version semantics.
- K2 supplies `WorkflowStateSnapshot`, `StateSequence`, and lifecycle transition controls.
- K3 supplies the upstream evidence that instances must preserve and consume.
- K5 supplies immutable workflow events that may later represent instance outcomes.

## CES Traceability
- `CES-B0-030.1`: instance meaning and approved-definition dependency.
- `CES-B0-030.4`: bounded workflow scope and tenant isolation.
- `CES-B0-030.17`: append-only audit evidence and replay inputs.
- `CES-B0-030.18`: stable identity, ownership, and deterministic execution invariants.

## API Implications
- K6 instance APIs should extend existing snapshot and workflow-reference patterns additively.
- No instance API may hide definition version, ownership path, or consumed upstream evidence.
- Instance construction and validation remain pure and side-effect free.

## Acceptance Criteria
- An instance can be described entirely through immutable identity, version, state, sequence, and evidence references.
- The instance model remains distinct from tasks, execution, and infrastructure concerns.
- The instance model reuses existing K1-K5 APIs without redesign.

## Final Determination
K6 workflow instances are immutable identity-bearing state records whose legal progression depends on approved definitions, bounded scope, and preserved upstream evidence.

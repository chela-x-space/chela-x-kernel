# K6.8 Workflow Failure And Recovery

## Status
Draft

## Purpose
Define bounded workflow failure, retry, and recovery behavior for K6.

## Scope
- Stable workflow failure classes.
- Retry-limit and recovery-path references.
- Recovery revalidation, bounded retry, and audit requirements.

## Out of Scope
- Task retry.
- Execution rollback.
- Schedulers, timers, queues, or infrastructure remediation.
- Unbounded loop handling outside deterministic validation.

## Canonical Terminology
- Workflow Failure Code: the stable English failure class for one workflow failure outcome.
- Workflow Retry: bounded re-entry under the same approved definition version.
- Workflow Recovery: the governed corrective path that returns a failed workflow to a valid executable state.

## Structures
- `WorkflowRetryLimit`.
- `WorkflowRetryPolicyReference`.
- `WorkflowRecoveryReference`.
- Existing K2 workflow failure-code and workflow-guard semantics.

## Invariants
- Failure codes are stable and deterministic.
- Deny-class failures block execution rather than degrade silently.
- Retry is bounded and definition-version bound.
- Recovery preserves scope and authority meaning and revalidates prerequisites before `Ready` or `Running`.

## Validation Precedence
1. Failure-state legitimacy and stable failure-code presence.
2. Retry-policy completeness and bounded retry-limit check.
3. Recovery-path declaration and corrective-path completeness.
4. Fresh policy, authorization, delegation, decision, scope, participant, and audit revalidation.
5. Unbounded-cycle rejection.

## Deterministic Outcomes
- Equivalent failure inputs produce equivalent failure, retry, and recovery outcomes.
- Exceeded retry limits or missing recovery paths reject deterministically.
- Recovery cannot silently widen scope, authority, or SoD meaning.

## K1-K5 Relationships
- K2 provides current failure-transition and recovery-guard enforcement.
- K3 provides the upstream evidence that must be revalidated on recovery.
- K5 provides immutable failure, retry, and recovery facts once accepted.

## CES Traceability
- `CES-B0-030.13`: stable workflow failure codes and deny-class blocking.
- `CES-B0-030.14`: bounded retry and fresh prerequisite revalidation.
- `CES-B0-030.17`: audit evidence for recovery and replay.
- `CES-B0-030.18`: no unbounded circular execution.

## API Implications
- K6 failure and recovery APIs should extend existing workflow reference types additively.
- Recovery validation should remain a pure function over explicit inputs.
- No recovery API may imply scheduler-driven retries or infrastructure-managed compensation.

## Acceptance Criteria
- Failure codes, retry limits, and recovery references remain stable and explicit.
- Recovery depends on fresh revalidation and bounded retry.
- K6 failure and recovery remains within workflow coordination and does not absorb task or execution recovery.

## Final Determination
K6 workflow failure and recovery is a bounded, evidence-driven correction model that preserves deterministic failure meaning, stable codes, and explicit revalidation before execution may resume.

# K6.7 Workflow Event Integration

## Status
Draft

## Purpose
Define how K6 consumes and emits enterprise events using the completed K5 event system.

## Scope
- Workflow consumption of accepted enterprise events as facts.
- Workflow production of immutable workflow facts after accepted definition or instance outcomes.
- Stream and replay relationships needed to preserve deterministic workflow meaning.

## Out of Scope
- Event Bus.
- Event Store implementation.
- Transport publication, subscriptions, or infrastructure handlers.
- Event-schema registries beyond existing K5 APIs.

## Canonical Terminology
- Workflow Event Fact: an immutable event describing an accepted workflow outcome.
- Workflow Event Replay: deterministic reconstruction of workflow meaning from workflow event history.
- Event Stream: the append-only K5 sequence that preserves one workflow replay history.

## Structures
- `EventEnvelope<P>`.
- `EventStreamId`, `EventReplayEntry<P>`, `EventReplayRequest`, and replay validation functions.
- Workflow event categories and traces already supported by `event.rs`, including workflow subject and trace references.

## Invariants
- Workflow consumes accepted facts; it does not treat requests as facts.
- Workflow-emitted events are append-only and replayable.
- Replay meaning is definition-version bound and input-order authoritative.
- Workflow event publication occurs only after prior K1-K6 validation succeeds.

## Validation Precedence
1. Workflow outcome acceptance under K2 and K3 guards.
2. Event-envelope completeness and event-domain validation.
3. Stream membership and append-order validation.
4. Replay ordering and requested-stream validation.
5. Workflow event replay consistency with workflow state meaning.

## Deterministic Outcomes
- Equivalent workflow outcomes produce equivalent event-validation outcomes.
- Equivalent replay histories reconstruct equivalent workflow state meaning.
- No event integration step mutates historical workflow facts.

## K1-K5 Relationships
- K5 is the direct event authority for workflow facts.
- K2 transition acceptance determines whether a workflow outcome may become an event.
- K3 evidence must be preserved into workflow audit and event traceability.
- K6 consumes K5 replay semantics; it does not redefine them.

## CES Traceability
- `CES-B0-030.11`: workflow event set and immutable-fact rules.
- `CES-B0-030.17`: audit replay and deterministic state reconstruction.
- `CES-B0-030.18`: append-only, deterministic workflow meaning.
- `07-workflow-architecture.md`: workflow consumes enterprise events and may later produce enterprise events through approved mechanisms.

## API Implications
- Future K6 event-integration APIs should compose existing K5 validators rather than duplicate them.
- Workflow event types should remain within the approved `workflow.*` event category surface.
- No K6 API may require Event Bus, Event Store, or transport concerns.

## Acceptance Criteria
- Workflow event integration uses only completed K5 contracts.
- Workflow facts remain immutable, append-only, and replayable.
- K6 event integration introduces no infrastructure dependency.

## Final Determination
K6 workflow event integration is a downstream composition of K2/K3 workflow outcomes with K5 event and replay contracts, preserving immutable facts and deterministic replay without introducing an Event Bus or Event Store.

## Purpose
Define deterministic task-to-task dependency semantics that contribute to readiness without executing or mutating tasks.

## Scope
- Dependency identity and direction
- Dependency type semantics
- Dependency validation and cycle detection where detectable
- Readiness contribution

## Non-Goals
- Temporal scheduling
- Automatic downstream execution
- Automatic state mutation
- Cross-workflow orchestration redesign

## Terminology
- `TaskDependency`: explicit dependency edge between task identities.
- `TaskDependencyId`: stable identity of one governed dependency edge.
- `TaskDependencyType`: declared type of dependency.
- `TaskDependencySource`: upstream task reference.
- `TaskDependencyTarget`: downstream task reference.
- `TaskDependencyStatus`: satisfied, unsatisfied, rejected, or unresolved dependency state.
- `TaskDependencyRequirement`: declared rule attached to a dependency.
- `TaskDependencyValidation`: deterministic validation outcome.
- `TaskDependencyGraphReference`: immutable reference to an evaluated dependency set.

## Domain Model
- Dependencies are explicit edges over stable task identities.
- Dependency identity remains explicit for audit, traceability, evidence association, dependency history, and deterministic reference.
- Dependency direction is from source to target.
- Dependency validation contributes to readiness only; it does not execute work.
- At minimum the specification evaluates completion, success, evidence, and output dependencies.

## Normative Requirements
- Dependency direction MUST be explicit.
- A dependency MUST reference stable task identities.
- Self-dependency MUST be rejected.
- Same semantic edge plus same `TaskDependencyId` MUST return deterministic no-op.
- Same semantic edge plus different `TaskDependencyId` MUST be rejected as duplicate-dependency conflict.
- Circular dependency MUST be rejected when the evaluated dependency set makes the cycle detectable.
- Dependency validation MUST be side-effect free.
- Satisfied and unsatisfied states MUST be distinguishable.
- Completion dependency MAY accept any terminal completion according to its declared rule.
- Success dependency MUST reject failed or cancelled upstream tasks.
- Output dependency MUST validate the declared output requirement.
- Evidence dependency MUST validate required evidence presence.
- Dependency satisfaction MUST contribute to readiness.
- Dependency creation MUST NOT execute tasks.
- Dependency no-op MUST NOT change dependency graph, readiness, lifecycle, assignment, or evidence.
- Dependency satisfaction MUST NOT mutate downstream state automatically.
- Workflow step ordering MUST not be redefined by K7 dependency semantics.
- Cross-workflow task dependencies MUST remain deferred unless explicitly authorized.

## Invariants
- Dependency direction is explicit.
- Dependency identity is stable.
- Readiness contribution is deterministic.
- Dependency evaluation is side-effect free.

## Validation Rules
- Missing source reference MUST be rejected.
- Missing target reference MUST be rejected.
- Self-dependency MUST be rejected.
- Semantic edge equality MUST mean same source, same target, same dependency type, and same declared requirement.
- Duplicate same-identity edge MUST return deterministic no-op.
- Duplicate different-identity semantic edge MUST return deterministic conflict rejection.
- Detectable cycles MUST be rejected.
- Missing referenced tasks MUST be treated as validation failure or unresolved reference, not infrastructure lookup.

## Rejection Conditions
- Self-dependency
- Duplicate semantic edge with different `TaskDependencyId`
- Detectable cycle
- Unsupported dependency type
- Missing referenced task in the explicit evaluation set
- Output or evidence requirement contradiction

## Compatibility Requirements
- K7 dependency coordination MUST remain compatible with K6 workflow step ordering.
- K7 MUST NOT redefine workflow orchestration through dependency semantics.
- K7 MUST remain compatible with frozen K5 evidence and event-reference patterns where dependency satisfaction consumes them by reference only.

## Public API Expectations
- K7 SHOULD expose explicit dependency objects and validation outcomes.
- Dependency validation SHOULD expose satisfied, unsatisfied, unresolved, rejected, or no-op outcomes through deterministic domain types.

## Error Expectations
- Structural dependency errors MUST be distinct from unsatisfied-but-valid dependency states.
- Cycle detection failures MUST be deterministic.

## Test Requirements
- Valid dependency construction
- Self-dependency rejection
- Duplicate same-identity edge no-op
- Duplicate different-identity edge conflict rejection
- Detectable-cycle rejection
- Missing task unresolved reference
- Success dependency rejection on failed upstream task
- Output dependency validation
- Evidence dependency validation
- No state mutation and no task execution

## Traceability

## Acceptance Criteria
- Dependency coordination is explicit, deterministic, and non-executing.
- Readiness contribution is defined.
- Workflow orchestration semantics remain unchanged.

## Deferred Concerns

- Large-graph performance considerations
- Multi-graph conformance policies

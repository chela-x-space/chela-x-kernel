# K6.1 Workflow Engine Foundation

## Status
Draft

## Purpose
Define the additive K6 workflow-engine boundary for CHELA-X Kernel under the frozen architecture.

## Scope
- Workflow identity, definition-version binding, ownership path binding, and instance meaning.
- Deterministic workflow coordination derived from approved definitions.
- Reuse of K1-K5 identifiers, lifecycle, authorization evidence, and enterprise-event APIs.

## Out of Scope
- Task ownership, assignment, or execution semantics.
- Scheduler, async runtime, database, transport, Event Bus, Event Store, UI, or infrastructure.
- Independent workflow authority, policy evaluation, or delegation redesign.

## Canonical Terminology
- Workflow Definition: one approved process structure bound to a stable version.
- Workflow Instance: one governed execution of one approved workflow definition version.
- Workflow Engine: the K6 coordination layer that requests work but does not perform business work.

## Structures
- `WorkflowId` remains the stable workflow identifier.
- `WorkflowStateSnapshot` preserves workflow identity, ownership path, definition version, lifecycle, and sequence.
- `WorkflowRetryPolicyReference`, `WorkflowRecoveryReference`, and `WorkflowAuditEvidenceReference` remain additive K6 foundation inputs.

## Invariants
- Workflow identity is stable and definition-version bound.
- Workflow consumes policy, authorization, delegation, and decision outcomes; it does not create authority.
- Workflow coordinates work but does not execute work.
- Workflow remains tenant-isolated, deterministic, and audit-evidence bound.

## Validation Precedence
1. Workflow identity and definition version.
2. Ownership path and tenant scope compatibility.
3. Definition approval and lifecycle eligibility.
4. Consumed upstream evidence completeness.
5. Audit-evidence requirements for decision-relevant transitions.

## Deterministic Outcomes
- Equivalent workflow definitions and evidence sets produce equivalent validation outcomes.
- Indeterminate or deny-class upstream outcomes block workflow progress.
- No hidden time, scheduler state, transport state, or storage state influences K6 domain meaning.

## K1-K5 Relationships
- K1 supplies `WorkflowId`, `StableVersion`, ownership primitives, and identifier semantics.
- K2 supplies `WorkflowState`, `WorkflowStateSnapshot`, `TransitionRequest`, and `validate_workflow_transition`.
- K3 supplies policy, authorization, delegation, decision, and SoD evidence consumed by workflow.
- K4 constrains agent-scoped participation through supervision and trust requirements without introducing runtime execution.
- K5 supplies immutable workflow events, stream identity, replay ordering, and replay validation as downstream enterprise facts.

## CES Traceability
- `CES-B0-030.1`: workflow identity, definition, and instance meaning.
- `CES-B0-030.12`: deterministic workflow validation gates.
- `CES-B0-030.18`: cross-cutting workflow invariants.
- `CES-ADR-0005`: Policy -> Delegation -> Workflow dependency direction.

## API Implications
- K6 implementation should extend `crates/kernel-domain/src/workflow.rs` and reuse `crates/kernel-domain/src/state.rs`.
- New K6 APIs must remain data-only or pure-validation APIs.
- Existing K1-K5 public APIs remain preserved and are reused additively.

## Acceptance Criteria
- K6 structures derive only from approved kernel architecture, existing kernel-domain APIs, and CES Chapter 30 workflow authorities.
- K6 documentation preserves the frozen workflow boundary and does not assign task or execution semantics to workflow.
- No K6 foundation item requires infrastructure, runtime orchestration, or public-API redesign.

## Final Determination
K6 foundation in Kernel is the deterministic workflow-coordination layer built additively on K1-K5 contracts and bounded by the approved architecture freeze.

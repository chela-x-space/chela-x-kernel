# K6.2 Workflow Definition

## Status
Draft

## Purpose
Define the canonical workflow-definition model required before any workflow instance may exist.

## Scope
- Workflow definition identity, namespace meaning, version binding, structure, and approval dependency.
- Definition-bound stage, transition, scope, participant, input, output, and recovery declarations.

## Out of Scope
- Instance execution history.
- Task creation semantics.
- Runtime scheduling, persistence, or external schema registries.

## Canonical Terminology
- Workflow Namespace: the stable English namespace for one workflow family.
- Approved Definition: the workflow definition revision that is eligible to produce instances.
- Definition Version: the immutable semantic version consumed by instances and replay.

## Structures
- Stable identifier: `WorkflowId`.
- Stable semantic version: `StableVersion`.
- Ownership and scope primitives from K1 and K2 remain authoritative.
- Approved-definition references are consumed by instance snapshots and transition requests.

## Invariants
- Every workflow definition has one stable identifier and one stable version.
- A workflow definition declares one approved state machine.
- Definitions are immutable after approval; revision produces a new definition version.
- No definition widens scope or authority beyond upstream policy and delegation constraints.

## Validation Precedence
1. Identifier and namespace completeness.
2. Definition version completeness and stability.
3. Ownership path and scope model compatibility.
4. Stage graph and state-machine completeness.
5. Participant, command, event, retry, recovery, and audit declarations.
6. Approval eligibility and replayability.

## Deterministic Outcomes
- Equivalent definition artifacts validate identically.
- Missing required structure yields deterministic rejection before downstream execution concerns.
- A non-approved definition cannot create an instance.

## K1-K5 Relationships
- K1 provides identity, version, and ownership value objects.
- K2 provides the allowed workflow lifecycle states and transition semantics.
- K3 provides the upstream policy, authorization, and delegation boundaries that definitions must consume.
- K5 provides event categories and replay semantics used by downstream workflow facts.

## CES Traceability
- `CES-B0-030.1`: workflow definition and identity requirements.
- `CES-B0-030.8`: ordered stage set and transition graph requirements.
- `CES-B0-030.9`: approved workflow state-machine requirements.
- `CES-B0-030.12`: workflow definition validation gates.

## API Implications
- K6 definition APIs should remain additive, immutable, and version-bound.
- Definition validation should reuse existing value objects instead of reparsing identifiers or versions.
- Definition approval must remain separate from instance execution.

## Acceptance Criteria
- The definition model identifies one workflow family and one approved revision unambiguously.
- The definition model stays within workflow coordination and does not absorb task or execution behavior.
- The definition model is representable with pure data and validation structures.

## Final Determination
K6 workflow definitions are immutable, approved, version-bound coordination contracts that govern instances without introducing independent authority or runtime execution behavior.

## Purpose
Define explicit accountable ownership and future-execution assignment semantics while keeping task governance distinct from execution.

## Scope
- Task ownership
- Task assignee binding
- Ownership and assignment authority
- Assignment statuses and deterministic reassignment outcomes
- Consumption of frozen K3 and K4 facts without redefining them

## Non-Goals
- Authorization-policy definition
- Agent supervision redesign
- Task execution
- Capacity reservation
- Scheduler behavior

## Terminology
- `TaskOwner`: accountable enterprise subject for governed outcome.
- `TaskOwnership`: explicit ownership binding for a task.
- `TaskOwnershipAuthority`: authority reference justifying ownership creation or transfer.
- `TaskOwnershipScope`: ownership scope carried from frozen upstream scope semantics.
- `TaskAssignee`: selected performer for a future execution layer.
- `TaskAssignment`: explicit assignment binding.
- `TaskAssignmentAuthority`: authority reference justifying assignment.
- `TaskAssignmentStatus`: deterministic assignment status only; it is not task lifecycle state.
- `TaskAssignmentReason`: stable reason for assignment or reassignment outcome.
- `TaskAssignmentEvidence`: immutable evidence supporting assignment.

## Domain Model
- Ownership and assignment are separate task concepts.
- Ownership is accountable governance.
- Assignment is future execution intent only.
- `Assigned` belongs only to `TaskAssignmentStatus`.
- Assignment MAY point to a human, agent, organizational unit, project, team, or other approved enterprise subject if represented by frozen upstream references.

## Normative Requirements
- Every active task MUST have exactly one effective owner unless a later approved specification explicitly permits an unowned draft state.
- Ownership MUST be explicit.
- Assignment MUST NOT imply ownership.
- Ownership MUST NOT imply assignment.
- An owner MAY be human, agent, organizational unit, project, or another permitted enterprise subject represented by existing frozen identifiers.
- An assignee MUST be eligible under existing K3 authorization and K4 registry or supervision information where applicable.
- K7 MUST consume authorization outcomes; it MUST NOT redefine policy.
- Assignment MUST reject ineligible, unavailable, deleted, suspended, or otherwise prohibited assignees when frozen upstream state exposes that status.
- Reassignment MUST preserve history through domain outcomes or references; it MUST NOT silently overwrite accountable history.
- Self-assignment MAY be allowed only where authority and eligibility permit.
- Multiple simultaneous assignees MUST NOT be introduced unless clearly approved by authoritative architecture.
- Assignment does not execute the task.
- Assignment MUST NOT change task lifecycle automatically.
- Task lifecycle transition MUST NOT change assignment automatically.
- A task MAY be `Pending` while assignment status is `Assigned`.
- A task MAY be `InProgress` while assignment status is `Accepted`.
- Revocation, release, rejection, and reassignment MUST be deterministic.

## Invariants
- Ownership and assignment remain distinct.
- One effective owner governs one active task.
- Assignment is explicit when present.
- Assignment status is distinct from task lifecycle state.
- Assignment does not imply execution.

## Validation Rules
- Missing owner on an active task MUST be rejected unless a later approved draft-state rule explicitly allows otherwise.
- Invalid ownership authority MUST be rejected.
- Invalid assignment authority MUST be rejected.
- Assignment to a prohibited upstream lifecycle or supervision state MUST be rejected when that upstream state is explicitly available.
- Unsupported multi-assignee semantics MUST be rejected.

## Rejection Conditions
- Missing effective owner for an active task
- Assignment without authority where authority is required
- Assignment to ineligible or prohibited assignee
- Silent reassignment without preserved history
- Unsupported simultaneous multiple assignees

## Compatibility Requirements
- Ownership and assignment MUST reuse K1 ownership and identity primitives where available.
- Assignment eligibility MUST consume K3 authorization decisions and K4 runtime or supervision facts without changing their semantics.
- K7 MUST NOT modify K6 workflow ownership or workflow-instance meaning.

## Public API Expectations
- K7 SHOULD expose separate ownership and assignment types or distinct fields with explicit read-only accessors.
- Assignment statuses SHOULD remain finite and deterministic, such as `Unassigned`, `Proposed`, `Assigned`, `Accepted`, `Rejected`, `Released`, `Revoked`, if later implementation confirms those are the minimal canonical set.

## Error Expectations
- Ownership errors MUST be distinct from assignment errors.
- Ineligibility, missing authority, and unsupported reassignment conditions MUST be distinguishable.

## Test Requirements
- Explicit ownership construction
- Assignment distinct from ownership
- `Assigned` used only as assignment status
- Lifecycle does not mutate assignment automatically
- Eligibility rejection
- Reassignment history preservation
- Deterministic assignment outcomes
- No execution side effect

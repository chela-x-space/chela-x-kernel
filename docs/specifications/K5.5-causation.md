---

# 1. Purpose

This specification defines the canonical Event Causation model for the
CHELA-X Enterprise Event System.

Causation identifies the accepted Enterprise Event that directly caused
another Enterprise Event.

Causation establishes deterministic lineage.

It does not establish operational grouping.

---

# 2. Objectives

The Causation model provides:

- deterministic lineage
- replay dependency
- enterprise traceability
- audit continuity
- historical reconstruction
- causal analysis

Causation never replaces Correlation.

---

# 3. Scope

This specification defines:

- Causation Identity
- Parent Event
- Child Event
- Causation validation
- Lineage rules

This specification does not define:

- Event grouping
- Event Streams
- Storage
- Transport
- Message delivery

---

# 4. Design Principles

Every Event may reference one direct parent Event.

A parent Event may have many child Events.

Causation is immutable.

Lineage is deterministic.

---

# 5. Canonical Model

The `causation_id` field is typed as `EventId`.

The Event System shall not define a separate `CausationId` identity
type.

Causation is a relationship between Events rather than an independently
identified enterprise entity.

Accepted Event A
        │
        ▼
Accepted Event B
        │
        ▼
Accepted Event C

Event B records:

    causation_id = Event A

Event C records:

    causation_id = Event B

The lineage forms a directed chain.

---

# 6. Root Events

A Root Event has no parent Event.

Examples include:

- System Startup
- External Request Accepted
- Scheduled Trigger
- User Command Accepted

Root Events omit `causation_id`.

---

# 7. Parent Rules

Every non-root Event shall reference exactly one accepted parent Event.

The parent Event shall:

- already exist
- possess a valid Event Identifier
- belong to canonical enterprise history

References to rejected or unknown Events are prohibited.

---

# 8. Validation Rules

Validation shall reject:

- self-causation
- circular lineage
- unknown parent
- invalid Event Identifier
- multiple parents
- mutable causation

Validation occurs before Event acceptance.

---

# 9. Architectural Constraints

Causation shall never:

- redefine Event Identity
- redefine Correlation
- modify Event Payload
- reorder Events
- bypass Event validation

Lineage preserves historical dependency only.

---

# 10. Relationship to Correlation

Correlation answers:

"What belongs together?"

Causation answers:

"What directly caused this Event?"

The two concepts are complementary.

Neither replaces the other.

---

# 11. Replay Foundation

Replay follows Causation order.

Parent Events shall be reconstructed before dependent child Events.

Replay behavior itself is specified in K5.8.

---

# 12. Canonical Determination

This specification establishes the canonical Event Causation model for
the CHELA-X Enterprise Event System.

Future Kernel capabilities shall preserve deterministic Event lineage as
defined in this specification.

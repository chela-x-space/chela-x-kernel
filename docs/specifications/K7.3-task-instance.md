## Purpose
Define the immutable governed task occurrence created from one task definition and explicit creation inputs.

## Scope
- Task-instance identity
- Definition snapshot binding
- Creation context
- Input and output bindings
- Workflow and step bindings
- Initial state compatibility with K7 lifecycle rules

## Non-Goals
- Execution
- Event publication
- Persistence
- Scheduler interaction
- Automatic workflow mutation

## Terminology
- `TaskInstance`: one concrete governed occurrence of a task definition.
- `TaskInstanceSequence`: deterministic sequence or ordering value for task-instance state history where required.
- `TaskDefinitionSnapshotReference`: immutable reference to the definition identity and version used at creation.
- `TaskCreationContext`: explicit creation inputs and governance context.
- `TaskInputBinding`: validated binding for required task inputs.
- `TaskOutputBinding`: validated output binding captured after accepted completion.
- `TaskWorkflowBinding`: optional workflow-instance or workflow-definition association.
- `TaskStepBinding`: optional workflow-step association.
- `TaskCreationAuthority`: authority reference used to justify task creation.

## Domain Model
- A `TaskInstance` MUST reference one `TaskDefinition`.
- A task instance MUST preserve the definition identity and version used at creation.
- A task instance MAY bind to a `WorkflowInstance` and `WorkflowStepReference`.
- A workflow association MUST remain additive and MUST NOT change K6 ownership or workflow semantics.

## Normative Requirements
- Every `TaskInstance` MUST reference a valid `TaskDefinition`.
- Every `TaskInstance` MUST have independent stable identity.
- A `TaskInstance` MUST capture the definition identity and version used at creation.
- Later definition changes MUST NOT silently alter an existing instance.
- Task creation MUST be deterministic from validated inputs.
- Creation MUST reject missing required inputs.
- Task instances MAY be associated with a `WorkflowInstance` and workflow step.
- Workflow association MUST NOT transfer task ownership automatically unless explicitly authorized by later specifications.
- Creating a `TaskInstance` MUST NOT execute it.
- Creating a `TaskInstance` MUST NOT publish events as an implicit side effect.
- Creating a `TaskInstance` MUST NOT persist it.
- Initial state MUST be explicit and governed by K7.6.
- Duplicate task identity MUST be rejected where detectable within the domain operation.

## Invariants
- Task-instance identity is distinct from task-definition identity.
- Definition snapshot is immutable after creation.
- Workflow or step binding is explicit when present.
- Task creation remains pure domain behavior.

## Validation Rules
- Missing definition reference MUST be rejected.
- Missing required creation inputs MUST be rejected.
- Unsupported workflow or step binding MUST be rejected.
- Self-inconsistent creation authority MUST be rejected.
- Duplicate identity within the evaluated domain operation MUST be rejected where detectable.

## Rejection Conditions
- Missing task-instance identity
- Missing definition snapshot reference
- Missing required input bindings
- Contradictory workflow or step binding
- Invalid initial state
- Detectable duplicate task-instance identity

## Compatibility Requirements
- Task-instance bindings MUST preserve K6 `WorkflowInstance` and `WorkflowStepReference` semantics.
- K7 MUST NOT reinterpret K6 workflow-state or step-coordination meaning.
- K7 MUST reuse K2 sequence and snapshot conventions where appropriate.

## Public API Expectations
- `TaskInstance` SHOULD follow immutable constructor-plus-getter conventions used by `WorkflowInstance`.
- Creation inputs SHOULD be explicit and validated at construction time.
- Workflow and step bindings SHOULD use read-only optional accessors.

## Error Expectations
- Creation failures MUST distinguish missing input, invalid workflow binding, invalid definition snapshot, and duplicate identity detection.

## Test Requirements
- Valid construction
- Definition snapshot preserved
- Workflow binding preserved
- Step binding preserved
- Required input rejection
- Deterministic creation
- No execution, no persistence, no event publication

- Definition snapshot binding is immutable.
